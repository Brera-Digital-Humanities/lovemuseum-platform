LoveMuseum API v1

Documentazione degli endpoint per app mobile LoveMuseum: autenticazione utenti, point of interest, foto, commenti, profili social e bookmark.

Base URL locale: http://localhost:8091 Prefix: /api/v1 Auth: Authorization: Bearer <token>

Formato

Le richieste possono essere inviate come form data oppure JSON. Le risposte sono JSON.

Codice Significato
200Operazione completata.
201Utente creato.
401Credenziali o token non validi.
403Operazione non permessa, utente non attivo o registrazioni disabilitate.
422Validazione fallita. La risposta contiene errors.
429Registrazione limitata dal throttle del plugin User.

Il campo birth_date usa il formato YYYY-MM-DD. La password minima segue la configurazione di Winter.User.

Modello Utente

Le risposte utente derivano da Winter\User\Models\User e rimuovono password, codici di reset/attivazione e persist code.

{
  "id": 1,
  "name": "Mario",
  "surname": "Rossi",
  "username": "mrossi",
  "email": "mario@example.com",
  "birth_date": "1990-05-20",
  "is_activated": true,
  "groups": ["registered"]
}
GET /api/v1/users/check

Verifica se username e/o email sono gia presenti.

Parametro queryObbligoNote
usernameUno tra username/email2-255 caratteri.
emailUno tra username/emailEmail valida, 6-255 caratteri.
 
curl "http://localhost:8091/api/v1/users/check?username=mrossi&email=mario@example.com"
{
  "username": {"exists": false, "available": true},
  "email": {"exists": false, "available": true}
}
POST /api/v1/users/register

Crea un utente frontend usando Auth::register() di Winter.User. Se l'utente non richiede attivazione, restituisce anche un JWT.

CampoObbligoRegole
nameSiStringa, 2-255.
surnameSiStringa, 2-255.
usernameSiUnivoco in users.username.
emailSiUnivoca in users.email.
birth_dateSiYYYY-MM-DD, precedente a oggi.
passwordSiConfermata da password_confirmation.
password_confirmationSiDeve combaciare con password.
 
curl -X POST http://localhost:8091/api/v1/users/register \
  -d name="Mario" \
  -d surname="Rossi" \
  -d username="mrossi" \
  -d email="mario@example.com" \
  -d birth_date="1990-05-20" \
  -d password="password123" \
  -d password_confirmation="password123"
POST /api/v1/users/login

Autentica un utente e restituisce un token JWT.

CampoObbligoNote
loginUno tra login/emailUsername o email, in base alla configurazione User.
emailUno tra login/emailAlternativa a login.
passwordSiPassword utente.
 
curl -X POST http://localhost:8091/api/v1/users/login \
  -d login="mario@example.com" \
  -d password="password123"
{
  "access_token": "eyJ0eXAiOiJKV1Qi...",
  "token_type": "bearer",
  "expires_in": 3600,
  "user": {}
}

Rotte protette

Tutte le rotte seguenti richiedono header Authorization: Bearer <token>.

GET /api/v1/users/logged

Restituisce il profilo dell'utente autenticato.

curl http://localhost:8091/api/v1/users/logged \
  -H "Authorization: Bearer TOKEN"

PATCH /api/v1/users/profile

Modifica i campi profilo. Accetta anche PUT.

CampoRegole
nameStringa, 2-255.
surnameStringa, 2-255.
usernameUnivoco, escluso l'utente corrente.
emailEmail univoca, escluso l'utente corrente.
birth_dateYYYY-MM-DD, precedente a oggi.
 
curl -X PATCH http://localhost:8091/api/v1/users/profile \
  -H "Authorization: Bearer TOKEN" \
  -d name="Mario" \
  -d surname="Rossi" \
  -d birth_date="1990-05-20"

POST /api/v1/users/refresh

Emette un nuovo JWT per l'utente autenticato.

POST /api/v1/users/logout

Con JWT stateless risponde {"success": true}. Il client deve eliminare il token locale.

Password

POST /api/v1/users/password/forgot

Invia la mail standard winter.user::mail.restore. Il link usa PASSWORD_RESET_URL se configurato, altrimenti APP_URL/password/reset.

curl -X POST http://localhost:8091/api/v1/users/password/forgot \
  -d email="mario@example.com"
POST /api/v1/users/password/reset

Imposta una nuova password usando il codice ricevuto via email.

curl -X POST http://localhost:8091/api/v1/users/password/reset \
  -d code="1!RESET_CODE" \
  -d password="newPassword123" \
  -d password_confirmation="newPassword123"

GET /api/v1/pois/categories

Lista categorie: id, name_en, name_it.

POST /api/v1/pois/create

Propone un nuovo luogo. Campi: lat, lon, name, category, descr, picture. Risponde {"status":"OK"} o {"status":"KO"}.

GET /api/v1/pois/{id}/view

Dettaglio POI con pictures, contatori, days_open, hours_open, tickets_info e booking_info.

Pictures

GET /api/v1/pictures/list

Lista ultime immagini pubblicate con oggetti relazionati poi e user, piu likes_num, bookmarks_num e comments_num.

GET /api/v1/pictures/{id}/view

Dettaglio immagine con likes, comments e bookmarks.

POST /api/v1/pictures/create

Crea una foto per un poi_id. Campi richiesti: poi_id, picture.

DELETE /api/v1/pictures/{id}/delete

Elimina solo se lo user proprietario della foto coincide con l'utente corrente mock.

Comments

Ogni commento contiene id, user, comment_text, comment_date, likes_num, likes, comments_num e comments per le risposte.

POST /api/v1/comments/{id}/create

Crea un commento o una risposta sul target indicato da {id}. Campo richiesto: comment_text.

DELETE /api/v1/comments/{id}/delete

Elimina solo se owner_user_id coincide con l'utente corrente mock.

PATCH /api/v1/comments/{id}/update

Aggiorna comment_text; accetta anche PUT.

Social Users

GET /api/v1/users/{id}/profile

Profilo pubblico con follower, following, contatori commenti, foto, bookmark e badge.

GET /api/v1/users/{id}/pictures

Foto caricate dall'utente.

GET /api/v1/users/my/pictures

Foto dell'utente corrente mock.

GET /api/v1/users/my/bookmarks

Bookmark dell'utente corrente, divisi in POI, foto e cartelle.

GET /api/v1/users/my/badges

Badge dell'utente corrente, inclusi quelli non ancora completati con progress e target.

POST /api/v1/users/folders/create

Crea una cartella bookmark. Campo richiesto: name.

PATCH /api/v1/users/folders/{id}/update

Modifica il nome di una cartella bookmark; accetta anche PUT.

DELETE /api/v1/users/folders/{id}/delete

Elimina una cartella bookmark dell'utente corrente mock.

GET /api/v1/users/folders/{id}

Restituisce tutti i bookmark contenuti nella cartella.
